## 一、单聊里反复形成的核心原则

### 1. 先确认真实日期，再确认最近交易日
A 股分析不能直接用“今天/昨天”做假设。单聊中已经明确过规则：

- 开盘期间，最近交易日通常就是当天。
- 周末或法定节假日，最近交易日是最近一个工作日。
- 做涨停复盘、日线选股、收盘后任务时，必须先确认交易日状态。

> 心得：日期错了，后面所有数据都会错；交易日判断是股票分析的第一道风控。

### 2. 真实数据优先，推演必须标注
单聊里多次出现“需要真实数据”“分析方向不对”的纠偏。由此沉淀出三层输出规范：

| 输出类型 | 使用条件 | 必须标注 |
|----------|----------|----------|
| 真实数据结论 | 已抓到行情、公告、新闻、K 线、数据库数据 | 数据来源、日期、是否收盘后 |
| 逻辑推演 | 数据源不可用或只有用户提供标题 | 写明“推演/待验证” |
| 操作建议 | 真实数据 + 逻辑 + 位置共同确认 | 买点、卖点、失效条件 |

### 3. 不懂代码时，AI 接管执行面
用户明确说过“看不懂代码，这个项目你来接管，深度集成进你的架构，你来用”。这意味着项目文档和脚本要服务自然语言工作流：

```text
用户给问题 -> AI 找数据/跑脚本 -> AI 解释结果 -> AI 归档方法论 -> 下次直接复用
```

## 二、数据源方法论

### 数据源分工

| 场景 | 首选 | 备用 | 备注 |
|------|------|------|------|
| 脚本分析 | AKShare | Baostock | AKShare 覆盖广，Baostock 适合历史日线 |
| 全市场收盘入库 | AKShare + 本地 SQLite | CSV 缓存 | 适合 15:00 后定时任务 |
| 盘中快速分析 | 单股/指数/板块轻量接口 | 缓存数据 | 避免全量接口超时 |
| 自动交易/Level-2 | QMT / MiniQMT | 券商终端 | 需要 Windows/券商权限 |
| 深度研究 | 萝卜投研、理杏仁、公告、新闻源 | 人工补充 | 适合基本面和事件验证 |

### 实战心得
- 全市场实时接口容易慢或超时，盘中不要把节奏卡死在全量扫描上。
- 收盘后适合跑全量并写入数据库，盘中适合轻量验证。
- 数据源失败时，不能硬编“确定结论”，只能输出待验证框架。

## 三、复盘报告工作流

单聊中围绕“涨停板块和个股梯队”“报告生成方法论”“更新 GitHub 项目”形成了稳定报告流程。

### 标准流程

```text
数据采集 -> 板块归因 -> 个股梯队 -> 交易剧本 -> 报告归档 -> 方法论更新
```

### 周报 8 章结构
1. 本周全景概览：指数、成交额、核心主线。
2. 指数日线复盘：逐日行情和关键转折。
3. 板块结构分析：领涨、领跌、抗跌、主线。
4. 个股点睛：龙头、中军、补涨、风险股。
5. 重大事件复盘：政策、产业、海外映射、公告。
6. 资金流向分析：成交额、融资、ETF、可得资金数据。
7. 交易宪法视角：因果性、阶段、卖点信号。
8. 下周展望：关键节点、两种剧本、操作原则。

### 报告质量底线
- 每个结论尽量有数据或事件支撑。
- 表格用于并列比较，段落用于解释因果。
- 不把“个股名单”当报告，必须有角色、逻辑、位置和风险。

## 四、选股模型沉淀

### 1. 涨停/巨量阳线 + 回踩 5 日线 + 缩倍量
单聊里反复要求寻找“涨停之后或者巨量阳线拉伸，回踩五日线缩倍量”的股票。模型条件可以固定为：

| 条件 | 含义 |
|------|------|
| 前期涨停或巨量阳线 | 证明主力活跃、题材被资金认可 |
| 连续回调不破趋势 | 洗盘而不是退潮 |
| 回踩 5 日线/10 日线 | 买点靠近支撑位 |
| 成交量缩到前高量的一半附近 | 抛压降低，筹码稳定 |
| 所在板块仍在行情内 | 避免只看形态忽略板块退潮 |

操作上只在分时企稳后低吸，跌破关键均线或 3 天不反包就离场。

### 2. 横盘震荡 + 资金流入 + MACD 向上
单聊里还要求找“横盘 2 个月/半年、持续资金流入、K 线健康、MACD 趋势交叉向上”的票。这个模型适合潜伏预期行情。

```text
横盘时间足够长 -> 筹码沉淀
资金持续流入 -> 有人提前布局
MACD 金叉或趋势上行 -> 技术开始转强
板块有未来事件 -> 有被点火的理由
```

关键是“在预期行情内”：没有未来事件的横盘，只是横盘；有事件的横盘，才可能变成启动前夜。

## 五、早评吸收与盘中推演

5 月 22 日单聊中，用户给了一段早评，随后衍生出特斯拉、大消费、主线/次线、科技线分化等分析。可沉淀为早评处理流程：

### 早评处理 SOP
1. 把早评拆成事件：政策、海外映射、行业消息、指数观点。
2. 把事件映射到板块：FSD、统一大市场、量子、商业航天、半导体等。
3. 给板块排主次：主线、次线、观察线。
4. 给个股排角色：龙头、中军、先锋、补涨、淘汰区。
5. 写出盘中剧本：冲高回落、探底回升、主线分歧、次线卡位。

### 两个高价值概念

| 概念 | 含义 | 动作 |
|------|------|------|
| 倒车接人 | 核心资产缩量回调，逻辑未破 | 等均线和分时承接低吸 |
| 倒车碾人 | 高位纯概念放量下跌，逻辑证伪 | 不抄底，先退出 |

## 六、产业链地图方法论

5 月 23 日单聊学习了“板块分类和产业链细分”，并在服务器生成了 `industry_chain_logic.json`。这个材料的价值在于：不要只按题材名找股票，要按产业链位置找股票。

### 产业链四层拆法

| 层级 | 关注点 | 适合找什么 |
|------|--------|------------|
| 上游 | 资源、材料、设备、核心零部件 | 涨价、国产替代、供给收缩 |
| 中游 | 制造、封测、系统集成、平台 | 订单、产能、技术突破 |
| 下游 | 应用、渠道、终端品牌 | 消费爆发、应用落地 |
| 配套 | 物流、检测、软件、服务 | 政策补短板、效率提升 |

### 使用方式
- 先定位事件影响哪一层，不要直接平铺股票。
- 强关联公司优先，弱映射只能观察。
- 同一事件里，上游常看涨价，中游看订单，下游看需求，配套看政策。

## 七、事件驱动框架

### 1. 公共卫生事件
埃博拉分析沉淀的关键是：疾病传播方式决定炒作路径。

```text
传播方式 -> 防控需求 -> 真实受益环节 -> 个股梯队 -> 持续性判断
```

埃博拉不等同于流感/新冠，不是全民囤药逻辑，而是疫苗、特效药、检测、防护和援助逻辑。

### 2. 供给侧冲击事件
山西矿难分析沉淀的关键是：市场炒的不是事故本身，而是供给减少。

```text
事故 -> 安监升级 -> 停产/减产 -> 价格预期 -> 龙头受益
```

判断时必须问：影响哪个品种？影响产能多大？价格有没有响应？公司是直接受益还是板块跟风？

### 3. 档期/消费事件
影视暑期档分析沉淀的关键是：档期题材按节奏炒，不按单日消息炒。

```text
定档 -> 宣发 -> 预售 -> 票房验证 -> 利好兑现
```

院线、发行、影视制作、AI 应用、广告营销要分开看，不要混成一个“传媒板块”。

## 八、单聊沉淀后的执行清单

1. 每次股票分析先确认真实日期和最近交易日。
2. 数据能抓就抓，抓不到就明确标注“推演”。
3. 输出必须包含：事件、板块、产业链、个股梯队、操作条件、失效条件。
4. 有价值的对话不要只留在聊天里，要归档到项目文档或脚本。
5. 已经能脚本化的能力，优先沉淀为工具；还需要人工判断的能力，沉淀为 SOP。
